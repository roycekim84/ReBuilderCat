# 냥냥 유적 복원단 게임 시스템 설계서

## 1. 문서 목적

이 문서는 **냥냥 유적 복원단(ReBuilderCat)**의 핵심 게임 시스템을 정의한다.

포함 범위:

- 복원 시스템
- 성공/실패 판정
- 균열 시스템
- 보상/재화/재료
- 유적/파트 진행도
- 역사 카드/도감
- 광고 보상과 IAP 연결 지점
- 로컬 저장 데이터 구조

이 문서의 목적은 구현 중 시스템이 과하게 커지는 것을 막고, MVP 범위 안에서 실제 출시 가능한 구조를 유지하는 것이다.

---

## 2. 핵심 시스템 요약

게임은 다음 3개의 핵심 값으로 돌아간다.

1. **성공률**  
   복원을 시도했을 때 성공할 확률이다.

2. **현재 보상**  
   지금 완료하기를 눌렀을 때 받을 수 있는 보상이다.

3. **균열 위험**  
   실패했을 때 손상이 발생할 위험도다.

유저는 이 3개를 보고 계속 복원할지, 현재 상태로 완료할지 결정한다.

---

## 3. 주요 용어

| 용어 | 의미 |
|---|---|
| 파트 | 아시아, 인도, 중동 등 월드맵의 큰 지역 |
| 유적 | 복원 대상이 되는 스테이지 |
| 복원 부위 | 유적 안에서 복원하는 개별 대상 |
| 복원 단계 | 부위의 현재 복원 레벨 |
| 복원 시도 | 복원하기 버튼을 누른 1회 행동 |
| 완료하기 | 현재 상태로 보상을 받고 복원 세션을 종료 |
| 균열 | 실패 시 발생하는 손상 상태 |
| 역사 카드 | 유적 완료 시 열리는 짧은 설명 카드 |
| 복원 재료 | 복원 시도에 필요한 기본 자원 |
| 골드 | 보상, 구매, 해금 등에 쓰이는 기본 재화 |

## 4. 데이터 구조 개요

MVP에서는 서버 없이 로컬 저장만 사용한다.

권장 저장 방식:

- Hive
- JSON seed data + Hive save data

데이터는 크게 두 종류다.

### 4.1 고정 데이터

앱에 포함되는 변경되지 않는 기본 데이터다.

- 파트 데이터
- 유적 데이터
- 복원 부위 데이터
- 역사 카드 데이터
- 아이템 데이터
- 밸런스 테이블

### 4.2 저장 데이터

유저 플레이에 따라 변하는 데이터다.

- 보유 골드
- 보유 복원 재료
- 유적별 진행도
- 부위별 복원 상태
- 열린 역사 카드
- 광고 제거 구매 여부
- 오늘 무료 보급품 수령 여부
- 광고 보상 사용 횟수

## 5. 파트 시스템

파트는 세계여행 진행의 큰 단위다.

초기 파트 구조:

1. Part 0. 작은 시작섬
2. Part 1. 아시아의 첫 유적
3. Part 2. 인도와 남아시아
4. Part 3. 사막과 중동의 고대 도시
5. Part 4. 지중해와 유럽 고대 문명
6. Part 5. 아프리카의 왕국과 사원
7. Part 6. 아메리카 고대 문명

MVP에서는 Part 0~2 일부까지만 구현해도 된다.

---

## 6. 유적 시스템

유적은 하나의 스테이지 단위다.

각 유적은 다음 정보를 가진다.

```dart
class Relic {
  final String id;
  final String regionId;
  final String name;
  final String inspiredBy;
  final String countryHint;
  final String shortDescription;
  final int unlockOrder;
  final int requiredRelicCompletedCount;
  final List<String> partIds;
  final String historyCardId;
  final String backgroundAsset;
  final String previewAsset;
}
```

MVP 구현에서는 실제 Dart 모델을 이 구조와 유사하게 만든다.

---

## 7. 복원 부위 시스템

각 유적은 3~5개의 복원 부위를 가진다.

초기 권장:

- 튜토리얼 유적: 2~3개 부위
- 일반 유적: 4개 부위
- 파트 마지막 유적: 5개 부위

복원 부위 예시:

```text
숲속 고대 사원
- 입구 계단
- 돌기둥
- 이끼 낀 벽화
- 중앙 제단
- 고양이 장식상
```

부위 데이터 예시:

```dart
class RelicPart {
  final String id;
  final String relicId;
  final String name;
  final int maxLevel;
  final int baseMaterialCost;
  final int baseReward;
  final int difficultyTier;
  final String assetPrefix;
}
```

---

## 8. 복원 단계

부위별 기본 최대 단계는 5단계로 시작한다.

| 단계 | 상태 | 설명 |
|---|---|---|
| 0 | 낡음 | 복원 전 |
| 1 | 정리됨 | 먼지 제거 |
| 2 | 안정됨 | 큰 균열 수리 |
| 3 | 문양 복원 | 장식과 무늬 복원 |
| 4 | 완성 | 부위 복원 완료 |
| 5 | 완전 복원 | 선택형 최고 상태 |

MVP에서는 maxLevel = 5를 기본으로 하되, 튜토리얼만 3~4단계로 낮출 수 있다.

---

## 9. 복원 시도 비용

복원하기 버튼을 누를 때마다 복원 재료를 소모한다.

기본 공식:

```text
시도 비용 = baseMaterialCost + currentLevel * costIncreasePerLevel
```

초기 예시:

| 현재 단계 | 비용 예시 |
|---|---:|
| 0 | 5 |
| 1 | 8 |
| 2 | 12 |
| 3 | 17 |
| 4 | 23 |

튜토리얼에서는 비용을 낮게 유지한다.

---

## 10. 성공률 시스템

성공률은 유적 난이도와 현재 단계에 따라 결정된다.

기본 테이블:

| 현재 단계 | 쉬움 | 보통 | 어려움 |
|---|---:|---:|---:|
| 0 → 1 | 95% | 92% | 90% |
| 1 → 2 | 85% | 80% | 76% |
| 2 → 3 | 72% | 66% | 60% |
| 3 → 4 | 58% | 50% | 44% |
| 4 → 5 | 45% | 38% | 32% |

초기 유적은 쉬움 또는 보통만 사용한다.

---

## 11. 복원 판정

복원하기 버튼을 누르면 다음 순서로 처리한다.

1. 복원 재료 충분 여부 확인
2. 복원 재료 차감
3. 보조 아이템 효과 적용
4. 성공률 계산
5. 랜덤 판정
6. 성공 또는 실패 결과 적용
7. 저장 데이터 갱신
8. UI 애니메이션 출력

의사 코드:

```dart
void attemptRestore() {
  if (!hasEnoughMaterial()) {
    showMaterialShortagePopup();
    return;
  }

  consumeMaterial();
  final rate = calculateSuccessRate();
  final roll = random.nextDouble();

  if (roll < rate) {
    applyRestoreSuccess();
  } else {
    applyRestoreFailure();
  }

  saveGame();
}
```

## 12. 성공 처리

성공 시:

- 복원 단계 +1
- 현재 보상 증가
- 연속 성공 횟수 +1
- 균열 상태 일부 완화 가능
- 유적 이미지 단계 변경

보상 증가 공식 예시:

```text
현재 보상 = baseReward * rewardMultiplierByLevel[currentLevel]
```

보상 배율 예시:

| 단계 | 배율 |
|---|---:|
| 0 | 0.5 |
| 1 | 1.0 |
| 2 | 1.6 |
| 3 | 2.4 |
| 4 | 3.5 |
| 5 | 5.0 |

---

## 13. 실패 처리

실패는 너무 가혹하지 않게 처리한다.

실패 시 가능한 결과:

- 균열 +1
- 현재 보상 10~25% 감소
- 일정 확률로 복원 단계 -1
- 연속 성공 횟수 초기화

MVP 추천 실패 규칙:

| 현재 단계 | 실패 결과 |
|---|---|
| 0~1 | 보상 감소만 발생 |
| 2~3 | 균열 +1, 보상 감소 |
| 4~5 | 균열 +1, 낮은 확률로 단계 -1 |

완전 파괴는 MVP에서 사용하지 않는다.

---

## 14. 균열 시스템

균열은 실패 누적 위험을 표현하는 값이다.

각 복원 세션은 균열 값을 가진다.

```text
crackLevel: 0~3
```

| 균열 단계 | 상태 | 효과 |
|---|---|---|
| 0 | 안정 | 불이익 없음 |
| 1 | 작은 균열 | 완료 보상 -5% |
| 2 | 큰 균열 | 완료 보상 -15% |
| 3 | 위험 | 완료 보상 -30%, 추가 실패 시 단계 하락 가능 |

균열은 광고 복구 또는 안전 아이템으로 낮출 수 있다.

---

## 15. 완료하기 시스템

완료하기는 현재 상태로 복원 세션을 종료하는 행동이다.

완료 시:

1. 현재 보상 계산
2. 균열 패널티 적용
3. 골드 지급
4. 부위 완료도 저장
5. 유적 전체 복원도 갱신
6. 조건 충족 시 다음 부위 해금
7. 조건 충족 시 유적 완료 처리

완료 보상 공식 예시:

```text
최종 보상 = 현재 보상 * 균열 패널티 * 이벤트 보너스
```

---

## 16. 유적 전체 복원도

유적 전체 복원도는 부위별 최고 완료 단계로 계산한다.

```text
유적 복원도 = 모든 부위 완료 단계 합 / 모든 부위 최대 단계 합
```

예:

- 부위 4개
- 각 maxLevel 5
- 총 최대 20
- 현재 완료 단계 합 12
- 전체 복원도 60%

---

## 17. 유적 클리어 조건

유적은 60% 이상 복원하면 클리어 처리할 수 있다.

| 복원도 | 상태 |
|---|---|
| 0~59% | 진행 중 |
| 60~79% | 클리어 |
| 80~99% | 우수 복원 |
| 100% | 완전 복원 |

월드맵 다음 유적 해금은 60% 클리어 기준으로 한다.

도감 완전 수집이나 별 3개는 100% 기준으로 한다.

---

## 18. 별 등급 시스템

유적별 별 등급은 복원도로 결정한다.

| 별 | 조건 |
|---|---|
| 1성 | 60% 이상 |
| 2성 | 80% 이상 |
| 3성 | 100% |

MVP에서는 단순히 표시만 하고, 후속 콘텐츠 해금 조건으로 과하게 사용하지 않는다.

---

## 19. 재화 시스템

### 19.1 골드

골드는 다음 용도로 사용한다.

- 일부 유적 해금
- 복원 도구 구매
- 보급품 구매
- 추후 꾸미기 구매

골드 획득처:

- 완료하기 보상
- 유적 클리어 보상
- 무료 보급품
- 광고 보급품

### 19.2 복원 재료

복원 재료는 복원하기 시도에 필요하다.

획득처:

- 유적 완료 보상
- 무료 보급품
- 광고 보급품
- 상점 구매

MVP에서는 재화를 너무 많이 만들지 않는다.

사용 재화:

1. 골드
2. 복원 재료

## 20. 아이템 시스템

MVP에서는 아이템을 최소화한다.

초기 아이템 후보:

| 아이템 | 효과 | 획득처 |
|---|---|---|
| 행운 붓 | 다음 복원 성공률 +5% | 상점, 보급품 |
| 안전 장갑 | 다음 실패 피해 감소 | 상점, 광고 보상 |
| 고양이 간식 | 소량 재료 보너스 | 무료 보급품 |

주의:

- 아이템을 너무 많이 만들지 않는다.
- 능력치 조합 시스템은 MVP에서 제외한다.
- 장비 강화 시스템은 넣지 않는다.

## 21. 역사 카드 시스템

유적이 60% 이상 복원되면 기본 역사 카드가 열린다.

100% 완전 복원 시 카드에 완전 복원 마크가 붙는다.

역사 카드 데이터 예시:

```dart
class HistoryCard {
  final String id;
  final String relicId;
  final String title;
  final String inspiredBy;
  final String regionText;
  final String periodText;
  final List<String> descriptionLines;
  final String funFact;
}
```

카드 문장 원칙:

- 짧게
- 쉽게
- 실제 유적 기반
- 게임 내 유적명과 실제 영감 유적명을 구분

## 22. 광고 시스템

광고는 게임 흐름을 끊지 않는 보상형 중심으로 설계한다.

### 22.1 보상형 광고 1: 균열 복구

발생 위치:

- 복원 실패 후

효과:

- 직전 실패 피해 일부 또는 전부 복구
- crackLevel -1 또는 실패 결과 무효화

제한:

- 부위당 1회 또는 하루 제한
- 모든 실패에 강제 노출 금지

### 22.2 보상형 광고 2: 보급품 받기

발생 위치:

- 상점 / 보급품 화면
- 재료 부족 팝업

효과:

- 복원 재료 지급
- 소량 골드 지급 가능

제한:

- 하루 횟수 제한 가능

### 22.3 전면 광고

MVP에서는 보류 가능.

넣는다면:

- 유적 하나 완료 후 낮은 확률
- 부위 완료 후 낮은 확률
- 광고 제거 구매 시 제거

절대 금지:

- 복원 버튼 몇 번마다 강제 광고
- 실패할 때마다 강제 광고
- 도감 진입 시 광고

---

## 23. IAP 시스템

MVP IAP는 광고 제거 하나만 우선한다.

상품명 예시:

- 후원자 패스
- 광고 제거 후원팩

혜택:

- 전면 광고 제거
- 보상형 광고 버튼 사용 시 광고 없이 보상 지급
- 후원자 배지
- 소량의 재료 보너스

저장 값:

```text
isAdRemoved: bool
```

구매 복원 기능은 필수다.

---

## 24. 일일 보급품 시스템

하루 1회 무료 보급품을 제공한다.

보상 예시:

- 골드 100
- 복원 재료 30
- 낮은 확률로 행운 붓 1개

저장 값:

```text
lastDailySupplyClaimDate: yyyy-MM-dd
```

날짜 기준은 기기 로컬 날짜를 사용한다.

## 28. 튜토리얼 흐름

튜토리얼은 짧아야 한다.

1. 작은 항구 사당 도착
2. 첫 부위 “낡은 돌계단” 선택
3. 복원하기 1회 강제 안내
4. 성공 연출
5. 성공률/보상 설명
6. 완료하기 안내
7. 역사 카드 획득
8. 월드맵 다음 유적 해금

튜토리얼 중 실패는 발생하지 않게 하거나, 실패해도 피해가 없게 처리한다.

## 29. 에러 처리

### 재료 부족

- 재료 부족 팝업 표시
- 상점 이동 또는 광고 보급품 제안

### 광고 로드 실패

```text
지금은 탐험대 지원을 받을 수 없어요.
잠시 후 다시 시도해 주세요.
```

### 저장 실패

```text
저장 중 문제가 발생했어요.
앱을 종료하기 전에 다시 시도해 주세요.
```

MVP에서는 저장 실패가 자주 발생하지 않도록 모든 판정 후 즉시 저장한다.

## 31. MVP 시스템 완료 기준

다음 조건을 만족하면 시스템 MVP는 완료다.

- 앱을 껐다 켜도 진행도가 유지된다.
- 복원 시도가 성공/실패로 정상 판정된다.
- 복원 재료가 소모된다.
- 완료하기 시 골드가 지급된다.
- 부위 진행도가 저장된다.
- 유적 복원도가 계산된다.
- 60% 이상 복원 시 다음 유적이 열린다.
- 역사 카드가 정상 해금된다.
- 무료 보급품을 하루 1회 받을 수 있다.
- 광고/IAP 서비스가 mock 상태로도 교체 가능하게 분리되어 있다.

## 32. 시스템 설계 최종 방향

냥냥 유적 복원단의 시스템은 단순해야 한다.

이 게임은 복잡한 RPG가 아니다. 장비, 스킬, 레벨, 캐릭터 성장, 수십 개 재화가 들어가면 핵심이 흐려진다.

핵심은 다음 한 문장이다.

> 성공률을 보고 한 번 더 복원할지, 지금 완료할지 고민하는 게임.

모든 시스템은 이 재미를 강화하는 방향으로만 추가한다.
